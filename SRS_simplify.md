## 1. 引言

### 1.1 编写目的
本文件用于明确系统应实现的业务能力、规则边界、质量目标和验收口径，作为研发与测试的统一基线。

### 1.2 系统目标
1. 为阿尔兹海默症患者提供从走失发现到任务闭环的全流程协同能力。
2. 让家属、路人、管理员在同一业务闭环中高效协作。
3. 通过AI能力提升寻回效率，同时确保安全、合规与可审计。

### 1.3 范围说明

范围内：
- 患者档案与标识管理
- 线索上报与研判
- 寻回任务管理
- 物资申领与标签生命周期管理
- 身份权限与审计治理
- AI辅助决策与海报能力

范围外：
- 硬件生产制造
- 医疗诊断结论
- 外部政务系统改造

### 1.4 术语说明

| 术语 | 说明 |
| :--- | :--- |
| 路人发现者 | 匿名主体，无需注册即可通过扫码或兜底方式上报线索 |
| 主监护人 | 对患者拥有最高管理权限的家庭成员 |
| 短码（Short Code） | 为患者生成的简短业务标识，印制于物理标签上，用于在二维码污损时供路人手动输入以提供线索，同时避免内部系统 ID 外泄。 |
| 走失状态（lost_status） | 患者当前走失语义状态枚举，取值 `NORMAL`（正常/未走失）或 `MISSING`（走失中）。 |
| 误报 | 任务关闭类型之一，表示本次走失事件不成立 |
| 存疑线索 | 被系统判为高风险无效、需管理员复核的线索 |
| 标签状态 | 防走失标签在业务中的状态，如未绑定、已分配、已绑定、挂失、作废 |

## 2. 业务角色与权限需求

| 角色 | 主要职责 | 权限边界 |
| :--- | :--- | :--- |
| 家属/监护人 | 建档、发起任务、关闭任务、查看进展、绑定标签 | 仅可操作已授权患者 |
| 路人发现者 | 匿名上报线索、提交位置和图片 | 不可查看敏感隐私，不可进入管理功能 |
| 普通管理员 | 复核线索、处理工单、进行协查 | 不可绕过审计，不可越权修改核心权限 |
| 超级管理员 | 高危治理、系统级应急处置 | 所有高危操作必须可追溯 |

## 3. 总体业务流程需求

### 3.1 主流程
1. 家属为患者建档并绑定标签。
2. 患者走失后，家属发起寻回任务。
3. 路人通过扫码或手动兜底上报线索。
4. 系统进行线索有效性判断并更新任务进展。
5. AI根据实时线索与历史信息给出协助建议。
6. 家属或管理员完成处置后关闭任务。
7. 系统完成通知、审计与结果沉淀。

### 3.2 异常流程
- 线索存疑时进入管理员复核，不得悬挂。
- 物流异常时进入异常处置分支，不得卡死在发货状态。
- 误报关闭不得污染长期经验样本。
- 被移除成员不得通过历史请求夺取主监护权限。

## 5. 关键业务规则

| 编号 | 规则 |
| :--- | :--- |
| BR-001 | 路人线索链路保持匿名，不得要求注册。 |
| BR-002 | 任务关闭只依赖当前登录的用户信息。 |
| BR-003 | 误报任务不得进入长期救援经验沉淀。 |
| BR-004 | 存疑线索必须可通过或驳回，不得长期悬挂。 |
| BR-005 | 主监护转移必须双阶段，且仅目标受方可确认。 |
| BR-006 | 被移除成员不得通过历史请求恢复高权限。 |
| BR-007 | 标签作废不等于患者走失，作废标签不得进入紧急上报链路。 |
| BR-008 | 物资异常处置必须可闭环，不得停留在中间态。 |
| BR-009 | 短码必须由服务端发号体系生成并唯一。 |
| BR-010 | 所有关键操作必须满足可审计、可追踪。 |

## 6. 数据需求（概念模型级）

### 6.1 核心业务实体

| 实体 | 关键业务属性（需求级） |
| :--- | :--- |
| 患者档案 | 姓名、年龄、短码、照片、健康相关描述、当前走失状态（lost_status: NORMAL/MISSING） |
| 监护关系 | 关系角色、关系状态、邀请与转移状态 |
| 标签资产 | 标签编码、标签类型、标签状态、关联患者、关联工单、作废原因 |
| 寻回任务 | 患者、任务状态、发起时间、结束时间、关闭类型、关闭原因 |
| 线索记录 | 标签、患者、位置、描述、图片、有效性、复核结果、驳回信息 |
| 物资工单 | 申领人、收货信息、工单状态、标签编码、物流信息、取消/驳回/异常原因 |
| AI会话 | 会话上下文、输入输出摘要、配额消耗、处理结果 |
| 审计日志 | 操作主体、操作对象、动作、结果、时间、追踪标识 |

### 6.2 数据一致性要求
1. 关键对象状态应在业务语义上保持一致，不得出现互相冲突。
2. 任何关键状态变化都必须可追溯到触发动作与责任主体。
3. 数据写入失败不得造成“用户看到成功但系统无记录”的结果。

## 7. 外部交互需求（能力级）

### 7.1 必要接口能力
- 任务发起与任务关闭
- 线索上报与线索复核
- 监护成员邀请、接受、移除、主监护转移
- 标签申领、发货、绑定、异常处理
- AI对话与建议返回

### 7.2 交互要求
1. 写操作必须具备幂等能力。
2. 所有关键链路必须具备统一追踪标识。
3. 匿名入口只暴露最小必要信息，不得暴露内部主键。

## 8. 非功能需求

| 类别 | 需求 |
| :--- | :--- |
| 可用性 | 系统应具备高可用能力，确保救援场景可持续服务。 |
| 性能 | 核心写入与告警链路应满足高并发下可接受响应。 |
| 可靠性 | 异常场景应可恢复，关键流程不因单点失败永久中断。 |
| 安全性 | 必须具备鉴权、风控、防重放与最小暴露能力。 |
| 隐私合规 | 公众链路不得泄露患者敏感信息。 |
| 可观测性 | 关键流程应可追踪、可审计、可复盘。 |

## 9. 验收标准（需求级）

1. 主流程可完成建档、发起任务、上报线索、处置关闭的闭环演示。
2. 路人可在匿名前提下完成上报，不需注册。
3. 同一患者不存在并发进行中的重复任务。
4. 主监护转移、撤销、拒绝、移除竞态均可被正确处理。
5. 存疑线索通过与驳回两条路径均可走通且有结果记录。
6. 误报关闭必须要求原因，且误报不会污染长期经验数据。
7. 物资流程可覆盖正常链路与异常链路，状态可闭环。
8. 标签状态语义清晰，走失与作废不混淆。
9. AI能力在限额、异常、降级场景下仍可给出可解释结果。
10. 关键操作具备审计证据并可按追踪标识检索。

2. 详细接口字段、状态机实现细节、数据库约束、消息投递机制应在详细设计文档中给出。
