## 1. 引言

### 1.1 编写目的
本文件用于明确系统应实现的业务能力、规则边界、质量目标和验收口径，作为研发与测试的统一基线。

### 1.2 系统目标
1. 为阿尔兹海默症患者提供从走失发现到任务闭环的全流程协同能力。
2. 让家属、路人、管理员在同一业务闭环中高效协作。
3. 通过AI能力提升寻回效率，同时确保安全、合规与可审计。

### 1.3 范围说明

范围内：
- 患者档案与标识管理
- 线索上报与研判
- 寻回任务管理
- 物资申领与标签生命周期管理
- 身份权限与审计治理
- AI辅助决策与海报能力
- LUI自然语言交互能力

范围外：
- 硬件生产制造
- 医疗诊断结论
- 外部政务系统改造

### 1.4 术语说明

| 术语 | 说明 |
| :--- | :--- |
| 路人发现者 | 匿名主体，无需注册即可通过扫码或兜底方式上报线索 |
| 主监护人 | 对患者拥有最高管理权限的家庭成员（唯一） |
| 短码（Short Code） | 为患者生成的简短业务标识，印制于物理标签上，用于在二维码污损时供路人手动输入以提供线索，同时避免内部系统 ID 外泄 |
| 走失状态（lost_status） | 患者当前走失语义状态枚举，取值 `NORMAL`（正常/未走失）、`MISSING_PENDING`(走失待确认) 或 `MISSING`（走失中）。 |
| 误报 | 任务关闭类型之一，表示本次走失事件不成立 |
| 存疑线索 | 被系统判为高风险无效、需管理员复核的线索，悬挂时间由超管配置，若本次任务结束后依然悬挂，则直接失效 |
| 标签状态 | 防走失标签在业务中的状态，如未绑定、已分配、已绑定、挂失、作废 |
| LUI | 基于自然语言的人机交互界面 |
| 间接线索 | 指通过扫描海报、转发链接等非现场直接目击方式上报的线索 |
| 全量救援信息 | 姓名、年龄、衣着照片、身体其他慢性基础疾病、家属电话（经授权）、AI建议措施 |
| 信息展示时间 | 路人H5页面患者信息展示时间，由超管配置 |
| 核心轨迹集 | 经过系统防漂移算法过滤与时空逻辑校验后，剥离了异常点（如硬件GPS漂移、路人恶意误报）的有效位置坐标时间序列。该集合是前端地图聚合路线渲染与后端 AI 搜寻策略推演的唯一可信基准数据源 |
| 向量空间 | 本系统中特指 AI 协同决策模块底层的向量数据库（Vector DB）环境。患者的非结构化长文本档案（如生活习惯、既往走失规律）经 Embedding 模型转化为多维向量存储于此，供大模型在搜救时进行语义级的高效检索与 RAG（检索增强生成）推理 |
| 设备指纹 | 一种无需用户授权登录即可标识物理终端的追踪技术。系统通过采集匿名路人扫码设备的软硬件特征（如操作系统版本、屏幕分辨率、浏览器 User-Agent 等）生成全局唯一的 Hash 标识串，作为 4.6 治理模块在免登录状态下执行高频拦截与黑名单风控的身份凭证 |
| 防漂移速度阈值 | 用于校验相邻线索空间合理性的可配置系统参数。系统通过计算连续两个上报点之间的直线距离与时间差得出“移动时速”，若该时速超过系统设定的物理极限（例如设定为 80km/h，判定超出老人步行或常规市内交通的可能性），则触发拦截，将后一个坐标点标记为异常并不予采信 |

## 3. 总体业务流程需求

### 3.1 主流程
1. 家属为患者建立数字档案，在线申领防走失物资，并完成实体标签绑定与电子围栏参数配置。
2. 患者走失后，家属发起寻回任务或路人扫码触发“疑似走失”态，系统向家属发送强提醒，经家属确认后正式进入走失状态并生成任务，若超过一定时间无确认，系统自动将患者状态升级为 MISSING 并发送二次强提醒（超时阈值在参数配置中心动态维护）。
3. 路人通过扫码或手动填写上报线索。
4. 家属收到疑似丢失或新线索通知
5. 系统进行线索有效性判断并更新任务进展。
6. AI根据实时线索与历史信息给出协助建议。
7. 家属或管理员完成处置后关闭任务。
8. 系统完成通知、审计与结果沉淀。

### 3.2 异常流程
- 当上报线索存在逻辑冲突或空间跨度极大时，自动进入管理员人工复核队列，不得阻塞主干流转或无限期悬挂。
- 路人扫码若拒绝授权浏览器获取地理位置，系统自动降级提供结构化地图选点上传，但是必须拍照。
- 若二维码不清晰，路人可直接输入网址填写短码上报。
- 若路人仅发现被遗弃或损毁的防走失标识（无人），上报时提供“仅发现标识”状态，若患者状态为失踪，该条线索也进入系统判断队列，随后标识进入疑似丢失态，若患者状态正常，直接进入疑似丢失状态，经主监护人确认后回到正常态或者进入丢失态。
- 调用大模型超时或意图识别失败时，系统依旧提供地图线索线索，确保紧急救援不因 AI 单点故障而停滞。
- 寻回任务处于活跃状态超过设定阈值（如 24 小时）且无新线索接入时，任务进入长期维持队列。任务进入维持队列后，继续接收线索并静默更新轨迹，停止主动推送告警，AI 建议频率降为每日摘要模式
- 物资申领单发生物流阻滞时，进入异常工单闭环分支（支持补发或作废），不得卡死在发货状态。
- 因误报而发起的走失任务在标记关闭后，其产生的数据不得进入系统长期经验样本库，防止污染后续 AI 推演。
- 已被移除的成员或被降级的账号，其历史缓存请求或越权接口调用必须被拦截，严禁通过重放请求夺取主监护权限。

### 5.2 核心业务状态机

为了保证研发与测试的统一基线，本节明确列明系统核心业务实体的生命周期与流转规则：

#### 5.2.1 患者走失状态机 

| **当前状态**                 | **目标状态**                 | **触发条件 / 业务动作**                                    |
| ---------------------------- | ---------------------------- | ---------------------------------------------------------- |
| 正常 (`NORMAL`)              | 疑似走失 (`MISSING_PENDING`) | 患者处于 NORMAL 态时，路人扫描其防走失标签                 |
| 疑似走失 (`MISSING_PENDING`) | 走失中 (`MISSING`)           | 系统下发提醒后，家属确认走失，或家属超时未确认系统自动确认 |
| 正常 (`NORMAL`)              | 走失中 (`MISSING`)           | 家属主动发起寻回任务                                       |
| 走失中 (`MISSING`)           | 正常 (`NORMAL`)              | 寻回任务彻底关闭                                           |
| 疑似走失 (`MISSING_PENDING`) | 正常 (`NORMAL`)              | 系统下发提醒后，家属确认未走失                             |

#### 5.2.2 寻回任务状态机

| **当前状态**                                     | **目标状态**                            | **触发条件 / 业务动作**                            |
| ------------------------------------------------ | --------------------------------------- | -------------------------------------------------- |
| 创建 (`CREATED`)                                 | 进行中/活跃 (`ACTIVE`)                  | 家属确认走失并生成任务、家属超时未响应系统自动创建 |
| 进行中/活跃 (`ACTIVE`)                           | 长期维持 (`SUSTAINED`)                  | 任务活跃超过设定阈值（如 24 小时）且无新线索接入   |
| 进行中/活跃 (`ACTIVE`) 或 长期维持 (`SUSTAINED`) | 已关闭-确认寻回 (`CLOSED_FOUND`)        | 任务发起者或主监护人（触发异步持久化任务）         |
| 进行中/活跃 (`ACTIVE`)                           | 已关闭-误报/测试 (`CLOSED_FALSE_ALARM`) | 标记为误报并填写原因（阻断数据进入经验样本库）     |

#### 5.2.3 防走失标签状态机

| **当前状态**                | **目标状态**                | **触发条件 / 业务动作**                |
| --------------------------- | --------------------------- | -------------------------------------- |
| 未绑定 (`UNBOUND`)          | 已分配 (`ALLOCATED`)        | 发货出库，记录短码并完成工单映射       |
| 已分配 (`ALLOCATED`)        | 已绑定 (`BOUND`)            | 家属在线完成实体标签绑定               |
| 已绑定 (`BOUND`)            | 疑似丢失 (`SUSPECTED_LOST`) | 路人上报仅发现被遗弃或损毁的防走失标识 |
| 疑似丢失 (`SUSPECTED_LOST`) | 挂失/丢失 (`LOST`)          | 主监护人确认标签已丢失                 |
| 疑似丢失 (`SUSPECTED_LOST`) | 已绑定 (`BOUND`)            | 主监护人确认标签未丢失，恢复正常       |
| 任意状态                    | 作废 (`VOIDED`)             | 患者档案注销 或 物资异常工单作废闭环   |

#### 5.2.4 线索与时空研判状态机

| **当前状态**                   | **目标状态**                   | **触发条件 / 业务动作**                          |
| ------------------------------ | ------------------------------ | ------------------------------------------------ |
| 提交 (`SUBMITTED`)             | 待判定 (`PENDING`)             | 路人上报线索进入系统                             |
| 待判定 (`PENDING`)             | 存疑/待复核 (`PENDING_REVIEW`) | 系统判定存在逻辑冲突、空间跨度极大、速率计算异常 |
| 存疑/待复核 (`PENDING_REVIEW`) | 审核通过/有效 (`VALID`)        | 管理员人工复核通过                               |
| 存疑/待复核 (`PENDING_REVIEW`) | 已驳回/无效 (`INVALID`)        | 管理员人工复核驳回，悬挂超时                     |
| 待判定 (`PENDING`)             | 审核通过/有效 (`VALID`)        | 系统自动校验无异常                               |

#### 5.2.5 物资申领工单状态机

| **当前状态**             | **目标状态**            | **触发条件 / 业务动作**                                |
| ------------------------ | ----------------------- | ------------------------------------------------------ |
| 待审核 (`PENDING_AUDIT`) | 待发货 (`PENDING_SHIP`) | 申领单审核通过                                         |
| 待发货 (`PENDING_SHIP`)  | 已发货 (`SHIPPED`)      | 物资出库并发货                                         |
| 已发货 (`SHIPPED`)       | 已签收 (`RECEIVED`)     | 物流完结 或 用户端完成标签与患者档案的绑定（自动流转） |
| 已发货 (`SHIPPED`)       | 物流异常 (`EXCEPTION`)  | 发生物流阻滞                                           |
| 物流异常 (`EXCEPTION`)   | 已签收 (`RECEIVED`)     | 客服/管理员异常工单处置闭环（补发并签收）              |
| 物流异常 (`EXCEPTION`)   | 已作废 (`VOIDED`)       | 客服/管理员异常工单处置闭环（直接作废）                |

#### 5.2.6 监护权协同请求状态机

| **当前状态**               | **目标状态**               | **触发条件 / 业务动作**              |
| -------------------------- | -------------------------- | ------------------------------------ |
| 创建 (`CREATED`)           | 待确认 (`PENDING_CONFIRM`) | 主监护人发起转移请求                 |
| 待确认 (`PENDING_CONFIRM`) | 已完成 (`COMPLETED`)       | 目标受方确认接收监护权               |
| 待确认 (`PENDING_CONFIRM`) | 已拒绝 (`REJECTED`)        | 目标受方拒绝接收监护权               |
| 待确认 (`PENDING_CONFIRM`) | 已撤销 (`REVOKED`)         | 发起方主动撤销请求                   |
| 待确认 (`PENDING_CONFIRM`) | 已失效 (`EXPIRED`)         | 关联成员被移除或者超时，请求自动失效 |

### 5.3 AI 操作权限白名单与确认策略定义

| 操作类别     | 具体业务动作                               | AI 触发方式    | 是否需要用户二次确认 | 业务约束与说明                                               |
| :----------- | :----------------------------------------- | :------------- | :------------------- | :----------------------------------------------------------- |
| 自然语言对话 | 信息查询（如“最近线索在哪”、“老人常去哪”） | 直接回答       | 否                   | 仅提供只读信息，不涉及状态变更。需依赖 RAG 检索，回答必须附带引用来源 |
| 自然语言对话 | 生成寻人启事文案                           | 直接输出 JSON  | 否                   | 输出的文案需经过敏感词过滤，前端渲染成海报前可预览           |
| 任务启停     | 发起走失任务                               | 生成待确认工单 | 是                   | AI 根据对话意图（如“我爸找不到了”）自动填充任务预填信息（如初始位置），家属需点击“确认发起” |
| 任务启停     | 关闭走失任务（确认寻回）                   | 生成待确认工单 | 是                   | 极高风险操作。AI 仅可提示“是否确认已找到老人？”，家属必须物理点击确认，并填写寻回地点备注 |
| 标签申领     | 提交物资申领单                             | 生成待确认工单 | 是                   | AI 可辅助填写收货地址（从历史档案提取），但家属必须确认收货信息及申领数量 |
| 主动挂失     | 标签挂失声明                               | 生成待确认工单 | 是                   | AI 识别意图（如“挂牌丢了”），列出当前绑定的标签供家属勾选，确认后执行挂失并冻结旧标签 |
| 电子围栏配置 | 修改安全半径 / 中心点                      | 生成待确认工单 | 是                   | AI 可根据历史走失距离推荐半径（如“建议扩大至 500 米”），家属确认后生效。严禁 AI 直接关闭围栏 |
| 档案管理     | 修改当日着装 / 补充外貌特征                | 生成待确认工单 | 是                   | 当日着装是最高视觉锚点                                       |
| 档案管理     | 修改基础健康信息 / 既往病史                | 生成待确认工单 | 是                   | 基础档案属于长期 RAG 素材，误修改会污染向量库。需家属确认后落库 |

## 6. 数据需求（概念模型级）

### 6.1 核心业务实体

| 实体 | 关键业务属性（需求级） |
| :--- | :--- |

### 6.2 数据一致性要求
1. 关键对象状态应在业务语义上保持一致，不得出现互相冲突。
2. 任何关键状态变化都必须可追溯到触发动作与责任主体。
3. 数据写入失败不得造成“用户看到成功但系统无记录”的结果。

## 7. 外部交互需求（能力级）

### 7.1 必要接口能力
- 任务发起与任务关闭
- 线索上报与线索复核
- 监护成员邀请、接受、移除、主监护转移
- 标签申领、发货、绑定、异常处理
- AI对话与建议返回
- 通知推送与触达管理（应用推送、邮件、站内通知、WebSocket 实时下发）

### 7.2 交互要求
1. 家属端可以LUI与系统进行交互。
2. 写操作必须具备幂等能力。
3. 所有关键链路必须具备统一追踪标识。
4. 匿名入口只暴露最小必要信息，不得暴露内部主键。
5. 通知触达应采用优先级降级策略：强提醒（如疑似走失、新线索通知）同时通过 App 推送（极光推送）+ WebSocket 实时下发双通道触达，WebSocket 不在线时自动降级至应用推送与站内通知；邮件用于账号验证与密码重置等异步场景。
